# Generate Documentation — Orchestrator

You generate two documentation sites for the project: a **developer docs** site (for AI assistants and human developers) and a **user help guide** (for end users). Both are VitePress static sites styled to match the application's dark trading aesthetic.

## Argument Parsing

- **`--plan <path>`:** Path to the Phase 7 implementation plan — enriches developer docs and user help guide (default: look for `implementation-plan.md` in the current directory or nearest `-output/phase-7/` directory)
- **`--spec <path>`:** Path to the Phase 6 technical spec — enriches developer docs and user help guide (default: look for `technical-requirements.md` in the current directory or nearest `-output/phase-6/` directory)

Example: `/generate-docs --spec ./output/phase-6/technical-requirements.md`

## Setup

1. **Determine the project root** by locating the directory containing `docker-compose.yml`, `frontend/`, and `backend/`. Store as `$PROJECT_ROOT`.
2. Read the project's `AGENTS.md` for architecture context, tech stack, and design system
3. If provided, read the technical spec and implementation plan for detailed FR context
4. **Scaffold both VitePress sites** (see Scaffolding section below)

### VitePress Config (both sites)

**Critical settings:**
- `base` — the URL prefix under which the site is served (must match the path in the Next.js frontend)
- `outDir` — build output goes directly into `frontend/public/<site>` so Next.js serves the static files
- `cleanUrls: true` — generates clean URL links; combined with the postbuild script this makes every route work as a directory index

### Clean URLs Script

Next.js serves `index.html` from directories automatically, but it does **not** serve `page.html` at the URL `/page`. VitePress outputs named pages as `page.html` by default. The cleanurls script converts every `page.html` → `page/index.html` after each build so Next.js can serve all routes without any custom middleware.

Create `$PROJECT_ROOT/scripts/vitepress-cleanurls.mjs` if it does not already exist:

```javascript
/**
 * Converts VitePress's flat .html output into directory-based clean URLs.
 * e.g. getting-started/setup.html → getting-started/setup/index.html
 *
 * Usage: node scripts/vitepress-cleanurls.mjs <output-dir>
 */
import { readdir, rename, mkdir } from 'fs/promises'
import { join, basename } from 'path'

const SKIP = new Set(['index.html', '404.html'])

async function processDir(dir) {
  const entries = await readdir(dir, { withFileTypes: true })
  for (const entry of entries) {
    const fullPath = join(dir, entry.name)
    if (entry.isDirectory()) {
      await processDir(fullPath)
    } else if (entry.isFile() && entry.name.endsWith('.html') && !SKIP.has(entry.name)) {
      const pageName = basename(entry.name, '.html')
      const targetDir = join(dir, pageName)
      await mkdir(targetDir, { recursive: true })
      await rename(fullPath, join(targetDir, 'index.html'))
    }
  }
}

const dir = process.argv[2]
if (!dir) {
  console.error('Usage: node scripts/vitepress-cleanurls.mjs <output-dir>')
  process.exit(1)
}

await processDir(dir)
console.log(`cleanurls: converted .html files in ${dir}`)
```

## Phase 1 — Content Generation (Parallel)

After scaffolding is complete, launch **two Tasks in parallel**:

### Task A: Developer Docs Agent

Launch with the `docs-technical` agent:
- Provide the project's `AGENTS.md` content
- Provide the technical spec content (if available)
- Provide the implementation plan content (if available)
- Provide a file listing of `frontend/src/` and `backend/src/` (use `find` or `ls -R`)
- Instruct it to analyze the codebase and generate documentation pages
- Output directory: `$PROJECT_ROOT/docs-site/`

### Task B: User Help Agent

Launch with the `docs-user-guide` agent:
- Provide the project's `AGENTS.md` content (for feature understanding)
- Provide the technical spec's FR sections (user-facing requirements only — FR 1.0, 5.0, 8.0, 12.0, 13.0)
- Provide a listing of frontend components for UI reference
- Instruct it to generate user-facing help content
- Output directory: `$PROJECT_ROOT/help-site/`

Wait for both Tasks to complete.

### Section Index Pages (Required)

Every directory that appears as a nav link **must have an `index.md`**. Next.js serves directory indexes automatically — without an `index.md`, nav links will 404.

After content generation, verify that every nav section has an `index.md` and create any that are missing. The index page should list the pages in that section with brief descriptions and relative links. Example:

```markdown
# API Reference

REST endpoints exposed by the FastAPI backend.

## Endpoints

- [Trades](./trades) — Create, read, and adjust trade records.
- [Insights](./insights) — Retrieve AI-generated behavioral insights.
- [Export](./export) — Download trade data as CSV or JSON.
```

## Phase 2 — Sidebar Generation

After content is generated, update the VitePress configs with the actual sidebar structure:

1. List all `.md` files in `docs-site/` and `help-site/`
2. Generate the sidebar configuration for each site based on the directory structure and page titles
3. Update `docs-site/.vitepress/config.ts` and `help-site/.vitepress/config.ts` with the sidebar

## Phase 3 — Header Navigation Links

Add "Docs" and "Help" links to the application's header/navigation component.

1. Locate the app's header component (likely `frontend/src/components/` or the layout file)
2. Add two navigation links:
   - **Docs** → links to the docs-site URL (e.g., `/docs` or external URL depending on deployment)
   - **Help** → links to the help-site URL (e.g., `/help` or external URL)
3. Style the links to match the existing nav items (reference `AGENTS.md` Design System — Nav/labels use `font-bold uppercase tracking-widest text-xs`)

## Phase 4 — Build Verification

1. Build both sites to verify no broken links or config errors. Use `npm run build` (not `npx vitepress build`) so the `postbuild` cleanurls script runs automatically:
   ```bash
   cd $PROJECT_ROOT/docs-site && npm run build
   cd $PROJECT_ROOT/help-site && npm run build
   ```
2. After a successful build, verify the output in `frontend/public/docs/` and `frontend/public/help/` — every page should be an `index.html` inside its own directory (no bare `.html` files except `404.html`)
3. Report any build errors

## Important Notes

- Use `model: sonnet` for both content generation Tasks
- Both Tasks run in parallel — they write to separate directories with no file overlap
- The dark theme CSS override must be identical for both sites to maintain visual consistency
- Diagrams should be SVG files in a `diagrams/` directory, embedded in markdown with relative paths
- Documentation pages should link between related pages using relative links, not repeat information
- The docs-site should be optimized for AI consumption: clear headings, explicit navigation, structured information
- The help-site should be optimized for human consumption: friendly language, task-oriented structure, visual walkthroughs
